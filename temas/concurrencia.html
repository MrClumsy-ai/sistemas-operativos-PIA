Control de concurrencia

Cuando una aplicación Android se inicia, el primer thread generado por su proceso es el main thread, también conocido como UI Thread, el cual es responsable de manejar toda la lógica de la interfaz de usuario. Este es el thread más importante de una aplicación. Es el responsable de manejar toda la interacción de usuario y también enlaza juntas todas las apartes móviles de la aplicación. Las tareas de larga duración invocadas en el UI serán con toda seguridad fatales para una aplicación, y aparecerá un diálogo ANR o Aplicación No Responde.

El framework HaMeR

El framework HaMeR permite a los threads en background enviar mensajes o postear runnables en el UI thread y también a cualquier otro MessageQueue de un thread por medio de handlers. HaMeR hace referencia a Handler, Message y Runnable. También hay otras clases importantes que operan junto con HaMeR: Looper y MessageQueue.

Arquitectura de Android y su papel en la concurrencia

La arquitectura basada en componentes de Android añade otra complejidad llamada fuga de memoria. Una fuga de memoria se produce cuando el objeto no puede ser liberado aunque ya no sea útil. En el peor de los casos, las fugas de memoria pueden provocar un fallo en la aplicación. Sin embargo, aunque las cosas no vayan tan mal, quedarse corto de memoria puede forzar recolecciones de basura más frecuentes que causan “jank”.

Desde la adopción generalizada de la arquitectura MVC, la cual es una interfaz de usuario de un solo hilo, el diseño estándar de la interfaz de usuario es una cola atendida por un único hilo, en el caso de Android es el hilo principal.
